# StepByハッカソン ロジック 発表原稿

## 用語メモ

- `粒度` の読み方は `りゅうど`

## 30秒版

StepByで実装している処理フローについて説明します．  
この処理フローでは，GitHub APIとOctokitでIssueなどの必要な情報を取得し，Geminiでタスクを初心者向けのステップに分解して，最後にUIへ変換します．  
この仕組みによって，Issueをそのまま渡すのではなく，初心者が次に何をすればよいか分かる小さなステップへ分解しています．  
その結果，初心者でもStep by Stepでプロジェクトに参加しやすくなっています．

## 少しゆっくり話す版

StepByで実装している処理フローについて説明します．  
まず，GitHub APIとOctokitで，IssueやREADMEなどプロジェクト参加に必要な情報を取得します．  
次に，その情報をGeminiに渡して，初心者向けに扱いやすいJSON形式へ構造化します．  
最後に，その構造化データをUIに落とし込むことで，極小ステップやドキュメント導線，コマンドコピペのような形で，わかりやすく提示しています．  
この共通パイプラインでやっているのは，Issueをそのまま読ませることではなく，初心者が次にやることを順番に理解できる形へ変換することです．  
特に重要なのは，GitHubの情報そのものが難しいというより，Issueの粒度がまだ大きいことなので，それを小さなステップに分解して，初心者でも参加しやすくしている点です．

## 発表で一番言う版

StepByの処理フローは，GitHubの情報取得，GeminiでのJSON構造化，初心者向けUIへの変換，の3段で統一しています．  
この共通フローを使うことで，Issueを初心者でも取り組める小さなステップに変換しています．

## コメント反映版

StepByで実装している処理フローについて説明します．  
GitHub APIとOctokitでIssueやREADMEなど必要な情報を取得し，Geminiで初心者向けのJSONに構造化して，それをUIに変換する流れで統一しています．  
今このスライドで強調したいのは，Issueを初心者向けの小さな行動単位に変換する部分です．  
重要なのは，Issueをそのまま渡すのではなく，初心者でもStep by Stepでプロジェクト参加できる形にすることです．
